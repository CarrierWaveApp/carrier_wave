# FT8 Digital Mode Support — Product Requirements Document

**Date:** 2026-02-25
**Status:** Draft
**Author:** Claude + jsvana

## Overview

Add full FT8 transmit and receive capability to Carrier Wave, enabling users to make FT8 contacts directly from their iPhone. FT8 is the most popular digital mode in amateur radio, accounting for the majority of HF contacts worldwide. It's especially valuable for POTA activators using QRP power and compromised antennas.

## Goals

1. Full TX + RX FT8 operation from the iPhone
2. Seamless integration with existing logging and sync pipeline (POTA/QRZ/LoFi)
3. Simplified mobile-first experience — simpler than WSJT-X, comparable to iFTx/FT Ham
4. Rock-solid decoder validated against real-world test recordings
5. Clear user guidance for no-CAT radio setup

## Non-Goals (Future Work)

- FT4 support (same architecture, different timing — easy to add later)
- CAT/rig control (iOS platform limitation)
- PSKReporter upload
- Multi-pass subtraction decoding
- A priori (AP) decoding
- Contest-specific exchanges
- GridTracker-style map display
- Worked-before highlighting

## FT8 Protocol Summary

FT8 ("Franke-Taylor design, 8-FSK modulation") transmits 77-bit messages in 15-second slots using 8-tone GFSK modulation. Signals occupy ~50 Hz bandwidth and are decodable down to -21 dB SNR.

### Key Parameters

| Parameter | Value |
|-----------|-------|
| Slot duration | 15.0 seconds |
| TX duration | 12.64 seconds |
| Symbol rate | 6.25 baud (160 ms/symbol) |
| Tone spacing | 6.25 Hz |
| Tones | 8 (0-7) |
| Occupied bandwidth | ~50 Hz |
| Payload | 77 bits + 14-bit CRC + 83-bit LDPC parity = 174 coded bits |
| Symbols | 58 data + 21 Costas sync = 79 total |
| Sample rate | 12,000 Hz |
| Decode threshold | -21 dB SNR (in 2500 Hz bandwidth) |

### Encoding Pipeline

```
77-bit message → CRC-14 → LDPC(174,91) → 58 three-bit symbols
→ Gray code mapping → Insert Costas sync arrays (pattern: 3,1,4,0,6,5,2)
→ 79 symbols → 8-GFSK modulation → audio
```

### Standard QSO Exchange

```
Slot  Caller (CQ)          Responder
 1    CQ K1ABC FN42         --
 2    --                    K1ABC W9XYZ EN37
 3    W9XYZ K1ABC -11       --
 4    --                    K1ABC W9XYZ R-07
 5    W9XYZ K1ABC RR73      --
```

Signal reports are in dB SNR (e.g., "-12"), not traditional RST (e.g., "599").

### FT8 Standard Dial Frequencies

| Band | Frequency (MHz) | Notes |
|------|-----------------|-------|
| 160m | 1.840 | Limited activity |
| 80m | 3.573 | Popular nighttime |
| 40m | 7.074 | Very popular |
| 30m | 10.136 | Popular |

| 6m | 50.313 | VHF |

## Session Experience

### Entry Point

User creates a logging session and selects **FT8** as the mode. The UI transitions from the manual logging interface to the FT8 session view. If POTA-activated, the park reference carries through from `ActivationMetadata`.

### Three Operating Modes

1. **Listen** — Decode only. Waterfall + decode list active. No transmission. Default on session start so user can verify audio routing works before transmitting.

2. **Call CQ** — Run mode for activators. Transmits CQ (or CQ POTA) every other slot. Auto-sequencer handles responses. Returns to CQ after each completed QSO.

3. **Search & Pounce** — Monitor decodes, tap a CQ station to reply. Auto-sequencer handles the exchange. Returns to listening after QSO completes.

Tapping a CQ in Listen mode automatically switches to S&P and begins the reply sequence.

### QSO Lifecycle

1. **Initiated** — User taps CQ (S&P) or station responds to CQ (Run)
2. **In progress** — Auto-sequencer exchanges messages (grid → report → R+report → RR73)
3. **Complete** — RR73 sent or received → QSO auto-logged
4. **Synced** — SyncRecords created, uploads through existing POTA/QRZ/LoFi pipeline

### Timing Display

The 15-second cycle is shown as a progress bar with countdown:
- **Blue** = Receiving (decoding in progress)
- **Orange** = Transmitting
- **Gray** = Idle (between cycles)

### Decode List Behavior

Each 15-second decode cycle produces a batch of decoded messages. The list shows:
- **SNR** (dB) — signal strength of the decoded station
- **Message text** — the full FT8 message
- **Color coding** — Green for CQ calls, red/highlighted for messages directed at user's callsign, default for other traffic
- **Bold** — new decodes from current cycle (fade to normal after one cycle)

Tapping a CQ message in the list initiates a reply. Tapping any other message is informational only.

## Technical Architecture

### Codec Layer (CarrierWaveCore SPM Package)

Wrap [ft8_lib](https://github.com/kgoba/ft8_lib) (Karlis Goba, MIT license) as a C target in CarrierWaveCore with a Swift overlay. ft8_lib is proven on embedded/microcontroller platforms, uses < 200 KB RAM, and handles both encoding and decoding.

```
CarrierWaveCore/
├── Sources/
│   ├── CFT8/                      ← C target (ft8_lib vendored)
│   │   ├── include/
│   │   │   └── module.modulemap
│   │   ├── ft8/                   ← ft8_lib source files
│   │   │   ├── encode.c/h         ← Message encoding
│   │   │   ├── decode.c/h         ← Signal detection + decoding
│   │   │   ├── ldpc.c/h           ← LDPC(174,91) codec
│   │   │   ├── crc.c/h            ← CRC-14 computation
│   │   │   ├── pack.c/h           ← 77-bit message packing/unpacking
│   │   │   └── constants.c/h      ← LDPC matrices, Costas pattern
│   │   └── common/                ← KISS FFT + utilities
│   ├── CarrierWaveCore/
│   │   ├── FT8Message.swift       ← Parsed FT8 message types
│   │   ├── FT8Encoder.swift       ← Message → tone array
│   │   ├── FT8Decoder.swift       ← Audio samples → [FT8DecodeResult]
│   │   └── FT8Constants.swift     ← Frequencies, timing constants
```

### Swift API

```swift
/// Parsed FT8 message
enum FT8Message: Sendable, Equatable {
    case cq(call: String, grid: String, modifier: String?)
    case directed(from: String, to: String, grid: String)
    case signalReport(from: String, to: String, dB: Int)
    case rogerReport(from: String, to: String, dB: Int)
    case roger(from: String, to: String)
    case rogerEnd(from: String, to: String)
    case end(from: String, to: String)
    case freeText(String)
}

/// Decode result with metadata
struct FT8DecodeResult: Sendable {
    let message: FT8Message
    let snr: Int           // dB signal-to-noise ratio
    let deltaTime: Double  // time offset from slot boundary (seconds)
    let frequency: Double  // audio frequency (Hz)
}

/// Decode a 15-second audio buffer
func decodeFT8(samples: [Float], sampleRate: Int = 12000) -> [FT8DecodeResult]

/// Encode a message to audio tones
func encodeFT8(message: String, frequency: Double = 1500.0, sampleRate: Int = 12000) -> [Float]
```

### Audio Engine (App Target)

```
CarrierWave/Services/
├── FT8AudioEngine.swift      ← AVAudioEngine management
├── FT8SessionManager.swift   ← QSO state machine + auto-sequencer
└── FT8WaterfallData.swift    ← FFT computation for waterfall display
```

**`FT8AudioEngine`** (actor):
- Manages `AVAudioEngine` with `.playAndRecord` category, `.measurement` mode (disables voice processing — critical for data modes)
- Input tap: captures audio buffers, resamples 48 kHz → 12 kHz (4:1 decimation)
- Output: plays FT8 tone arrays through speaker/audio interface
- Route change detection: handles USB audio interface connect/disconnect
- Background audio: keeps audio session active when app is backgrounded

**`FT8SessionManager`** (@MainActor, @Observable):
- 15-second slot timer synchronized to UTC (iOS NTP clock)
- QSO state machine: `idle → calling → reportSent → reportReceived → complete`
- Auto-sequencer: determines next TX message based on QSO state and received messages
- Decode processing: updates decode list, detects responses to our CQ/call
- QSO completion: creates QSO + SyncRecords via existing model creation pattern
- Duplicate tracking: won't start new QSO with station already worked this session

**`FT8WaterfallData`** (@MainActor, @Observable):
- Receives audio buffers from audio engine
- Computes spectrogram: 1920-sample FFT at 12 kHz → 6.25 Hz frequency bins
- Provides 2D magnitude array for SwiftUI waterfall view
- Rolling window showing last ~60 seconds (4 slots)

### Data Flow

```
Microphone/Interface → AVAudioEngine input tap
    → Resample 48 kHz → 12 kHz (decimate by 4)
    → Accumulate 15 seconds of samples (180,000 samples)
    → FT8Decoder.decode() on background thread
    → [FT8DecodeResult] → FT8SessionManager (MainActor)
        → Update decode list (drives UI)
        → Check for response to our TX → advance QSO state
        → If QSO complete:
            → Create QSO model (callsign, band, mode="FT8", dB reports, grid)
            → Create SyncRecords for POTA/QRZ/LoFi
            → Park reference from ActivationMetadata (if POTA)

FT8SessionManager TX decision → FT8Encoder.encode(message)
    → [Float] audio samples at 12 kHz
    → Upsample to 48 kHz
    → AVAudioEngine player node → speaker/interface
    → Radio VOX triggers → on-air transmission
```

### Integration with Existing Models

**QSO model fields for FT8:**
- `callsign`: Other station's callsign (from FT8 message)
- `band`: Derived from selected frequency (existing BandUtilities)
- `mode`: "FT8"
- `rstSent`: dB report as string, e.g., "-12" (not traditional "599")
- `rstReceived`: dB report as string, e.g., "-07"
- `gridSquare`: From FT8 exchange (4-character Maidenhead)
- `frequency`: Dial frequency in MHz (from band selector)
- `timestamp`: UTC start of QSO
- `parkReference`: From `ActivationMetadata` (if POTA session)

**SyncRecords** created automatically for enabled destinations, exactly as manual QSOs.

**ADIF export** uses `MODE=FT8`, `RST_SENT=-12` (dB format). The existing ADIF exporter must accept string RST values that aren't traditional "599" format.

## Audio Setup & User Instructions (No CAT)

### First-Time Setup Wizard

When user first selects FT8 mode, show a guided setup:

**Step 1: Audio Connection**

Three options presented as cards:

1. **USB Audio Interface** (recommended) — Digirig, SignaLink, etc. Connect to iPhone via USB-C. Best quality, hardware VOX support. Instructions: plug in interface, connect to radio data port, Carrier Wave auto-detects USB audio device.

2. **Audio Cable (TRRS)** — Direct 3.5mm connection from radio to iPhone headset jack. Requires USB-C to 3.5mm adapter on modern iPhones. Radio must have VOX enabled. Note about optional 2.2k resistor for impedance matching.

3. **Speaker/Mic (Air)** — Place phone near radio speaker. No cables needed. Requires quiet environment and sensitive VOX setting. Suitable for quick portable operation only.

**Step 2: Radio Configuration**

Band-specific checklist:
1. Set mode to **USB** (Upper Sideband)
2. Tune to the FT8 frequency for selected band (highlighted in table)
3. Set power to **5-25 watts** (full duty cycle — don't run high power)
4. Enable **VOX** with fast release
5. Turn **AGC off** or slow
6. Widen **SSB bandwidth filter** to maximum (2.4-3.0 kHz)

**Step 3: Audio Level Verification**

Before enabling TX:
1. Start in Listen mode — verify decodes appear (confirms RX audio path)
2. For TX: brief test tone to verify radio ALC meter barely moves
3. Save connection method and band preference for next session

### Persistent Band Reference

The band selector always shows the expected dial frequency. When changing bands, a reminder banner appears: "Tune radio to XX.XXX MHz USB". This compensates for the inability to verify radio frequency programmatically.

### VOX Timing

The app handles VOX timing by:
- Starting tone generation ~100ms before slot boundary (pre-roll for VOX engage)
- Brief carrier tone at the start ensures VOX activates before Costas sync
- Clean taper at end for smooth VOX release
- The ~2.36-second gap between transmissions is sufficient for VOX dropout

## Testing Strategy

### Test Data

Downloaded and available in `docs/plans/ft8-samples/`:
- 3 WSJT-X official recordings (real off-air FT8, 12 kHz mono WAV, 15 seconds each)
- 7 ft8_lib test vectors (WAV + expected decode output .txt files)
- 10+ encode/decode round-trip test cases from ft8_lib test.c

### Test Layers

**Layer 1: Codec Unit Tests (CarrierWaveCore, no simulator required)**

`FT8EncoderTests`:
- Encode known messages → verify tone array matches expected
- Round-trip: encode → decode → verify message matches
- Edge cases: max-length callsigns, negative dB, CQ modifiers (POTA, DX, NA)
- Free text encoding/decoding
- Nonstandard callsign handling (EA8/G5LSI, compound calls)

`FT8DecoderTests`:
- Decode each ft8_lib test vector WAV → compare against .txt expected output
- Verify SNR, frequency, time offset within tolerance (±1 dB, ±10 Hz, ±0.1s)
- Multi-signal decode (busy band samples: expect 15+ signals per 15s window)
- Decode WSJT-X official recordings (smoke test: verify non-zero decode count)
- Edge cases: empty audio, noise-only audio, clipped audio (should return empty, not crash)

**Layer 2: Message Parsing Tests (CarrierWaveCore)**

`FT8MessageTests`:
- Parse all standard message types
- CQ modifiers: `CQ DX`, `CQ POTA`, `CQ NA`, `CQ 123`
- Callsign extraction from each message type
- Grid square extraction and validation
- Signal report extraction (positive and negative dB values)
- Free text handling

**Layer 3: Session State Machine Tests (App target)**

`FT8SessionManagerTests`:
- State transitions: idle → calling → reportSent → reportReceived → complete
- Auto-sequencer generates correct next TX message per state
- CQ mode: resumes CQ after completed QSO
- S&P mode: returns to idle after completed QSO
- Timeout: marks QSO abandoned after N cycles with no response
- Duplicate: won't initiate with already-worked station
- QSO completion creates QSO model with correct field values

**Layer 4: Audio Pipeline Tests**

`FT8AudioEngineTests`:
- Resampling: 48 kHz → 12 kHz decimation produces correct output
- Tone generation: encoded message produces valid audio samples
- Full pipeline: generate tones → decode → verify round-trip

**Layer 5: Integration Tests**

`FT8IntegrationTests`:
- Full flow: WAV input → decode → auto-sequence → QSO creation → SyncRecord
- POTA context: QSO inherits park reference from ActivationMetadata
- Signal report format: "-12" stored correctly in rstSent/rstReceived
- ADIF export: MODE=FT8, dB report format in RST fields

1. **Logging integration** — Auto-sync to POTA/QRZ/LoFi. No export/import needed.

## Known Limitations

1. **No CAT control** — iOS platform limitation. User must manually tune radio. Shared with all iOS FT8 apps.
2. **VOX-only TX** — Small delay at transmission start, mitigated by pre-roll tone.
3. **Single-pass decoding (v1)** — No multi-pass signal subtraction. Fewer decodes on very crowded bands vs WSJT-X. Adequate for typical operation.
4. **Clock dependency** — FT8 requires ±0.5s UTC accuracy. Relies on iOS NTP sync. Will display UTC and warn if drift detected.
5. **No split operation** — Cannot programmatically set separate TX/RX frequencies on radio. TX audio offset can be varied within the passband as a workaround.
